我原本设想的极其简单，在模型上做优化，直接把稀疏特征向量当压缩后的表示存起来，不还原成文本。下游消费时把特征向量注回同一模型的对应层，跳过token解码和重新编码的过程。好了，这是模型层的，那我写这个显然不是模型层的，因为这个我没个一个月我做不明白，但是我想到一个现在可以用，并且可以炒的方案，下面是我和alam聊天记录的ai总结

设计思路总览 · 草稿

摘要

本文梳理了一次从"文本语义压缩器"出发、逐步推演至"LLM 上下文管理基建层"的设计探索过程。讨论涵盖模型选型、数据构造、工程架构、召回机制、语义结构维护等核心议题，最终将问题定性为一个以动态语义数据库为核心的工程任务，而非单纯的模型训练任务。

1. 问题起点：文本语义压缩

初始目标是构建一个能够保留核心语义、精简文字表达的压缩器。区别于关键词抽取或抽象摘要，语义压缩的核心约束是：压缩后的文本必须在语义层面等价于原文，适合作为信息密度更高的替代输入。

初步方案：微调一个小模型专门承担此任务，以替代调用 SOTA 大模型的高成本路径。基座模型选定 Qwen2.5-1.5B-Instruct，训练数据通过大模型蒸馏生成，训练框架采用 LLaMA-Factory + QLoRA，运行于单张消费级 GPU（3090/4090）。

2. 流水线设计：传统切割 + 语义状态传递

为处理长文本，提出分块压缩流水线：

  • 传统方法负责粗粒度切割
  • 小模型对每个块执行语义压缩
  • 块间通过"语义 prompt"传递上下文状态，保证跨块连贯性

该设计的核心挑战在于：切割边界与语义单元不一致时，语义 prompt 的补偿能力有限；同时，语义状态的表达形式（摘要、实体列表、结构化描述）直接影响压缩质量和训练数据设计。

3. 与 SOTA 的对比分析

与 SOTA 大模型（GPT-4o、Claude 3.5 Sonnet 等）相比，本方案在通用压缩质量上存在差距，根本原因在于 SOTA 模型能够整体把握全文语义，而分块方案存在信息局部化损耗。

然而，评估框架的选择改变了竞争格局。当目标用途从"人类可读摘要"切换为"下游 LLM 的输入"，SOTA 的优势被大幅压缩——现有 SOTA 模型并未针对"压缩后下游任务表现"这一目标专门优化。若将训练目标直接对准下游任务准确率，小模型理论上可以在此窄任务上超越通用 SOTA。

4. 真实需求：LLM 长上下文压缩

真实使用场景锚定于 Claude Code 中 Claude Opus 级别模型的上下文溢出问题。复杂编程任务会快速耗尽上下文窗口，现有解法（直接调用 SOTA 压缩）成本高且效果一般。

关键洞察：上下文溢出的根本问题并非长度，而是噪声——对话历史中充斥着已被推翻的方案、重复的报错信息、以及不再相关的中间状态。因此，压缩器的核心价值是去噪，而非单纯缩短。

此外，若压缩后的上下文能让下游模型在降低 token 消耗的同时提升任务完成率，则该方案的定位将从"低成本替代品"升级为"质量更优的上下文管理层"。

5. 工程任务而非模型任务

随着场景深入，问题的本质发生转变。模型微调只是整个系统的一个算子，真正的工程复杂度集中在：

  • 如何接入 Claude Code 的上下文管道
  • 两类内容（对话历史 vs. 代码文件）的差异化处理逻辑
  • 两条管道之间的协调层设计
  • 整个链路的延迟预算控制

对话历史遵循时序衰减逻辑：早期探索过程可大幅压缩，近期操作和当前错误现场须完整保留。

代码文件遵循任务相关性逻辑：不按时间裁剪，而是根据当前任务保留接口定义和关键函数，折叠无关实现细节。

两条管道并非独立，对话历史的语义状态需反向指导代码文件的压缩粒度。

6. 上下文数据库：核心抽象的涌现

将"压缩器"扩展为"上下文数据库"，是本次讨论中最关键的概念跃迁。

现有 LLM 上下文的根本缺陷在于：线性结构、易失性、无组织——四类语义层次（事实性内容、操作记录、意图、状态）平铺在同一序列中，缺乏差异化管理。

上下文数据库的核心原语：

操作 语义
写入 内容 + 任务上下文 → 压缩 + 语义向量 + 元数据（来源、时间、重要性权重）
召回 当前任务 → 相关性排序 → 按 token 预算动态组装上下文窗口
更新 同一文件新版本进入时，旧版本降权或失效，而非简单覆盖
遗忘 长期未被召回的内容进一步压缩或归档，而非删除

与现有方案（RAG、MemGPT 等）的本质区别在于：本方案在基建层统一处理压缩、索引、召回和生命周期管理，而非在应用层做局部优化。

7. 召回问题：最难的部分

召回的根本难点在于：相关性是语义的，不是结构的。函数 B 与函数 A 的强耦合可能完全不体现在调用图中，而是隐含在业务逻辑里。规则无法发现这类关联，LLM 实时判断又带来延迟和成本问题。

出路：将实时语义判断转移至写入时的离线预计算。

具体路径：

  1. 静态分析建立调用图（规则层）
  2. LLM 扫描代码，识别调用图中未体现但语义上强耦合的关系，补充边（语义增强层，离线执行）
  3. 召回时走图遍历，从当前任务节点按权重扩散，按 token 预算截断

LLM 的职责因此转变为异步维护语义关系图，而非实时参与每次召回决策。

8. 已知问题

这个设计有真实的问题。我知道它们。有些我正在解决；有些我不打算解决，当作可接受的降级处理。

离线语义图会过时。 在任何真实代码库中，单个接口变更可能使图的大部分失效。“离线"和"保持最新"是互斥的——如果图不刷新，召回层就在消费过期关系并在下游产生幻觉；如果刷新，它就不再是离线的，成本优势消失。我不解决这个问题。回退方案是缩小图的范围：稳定的核心模块获得完整语义扫描，频繁变更的模块降级为纯向量检索并接受召回质量损失。图成为部分索引，而非完整索引。

双管道耦合是脆弱的。 让对话历史的语义状态反向传播以指导代码文件压缩，产生了级联依赖。如果意图摘要漂移，下游代码压缩就会系统性地做出错误决策——而且错误是静默的，只有在下游模型在实际任务上失败时才会浮现。我正在解决这个问题。方向是强制解耦：两个管道共享一个共同的任务状态表示作为输入，但都不将对方的输出作为依赖。协调发生在上游的共享状态层，而不是管道之间的运行中。

通用性主张为时过早。 当前设计在多个方面都是 Claude Code 特定的——AST 解析、调用图、符号级差异对比都是代码原生的。声称其他场景"自然适配"忽略了内容结构和生命周期的真实差异。目前，Claude Code 是唯一的验证目标。通用性是后来的问题。
